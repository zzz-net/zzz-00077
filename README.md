# 告警回放工具

本地仓库告警事件回放与演练系统，用于模拟和演练告警事件处理流程，验证告警规则有效性，培训操作员的事件响应能力。

## 功能特性

- ✅ **事件时间轴导入** - 支持JSON格式事件时间轴导入，附带乱序样例事件
- ✅ **告警规则配置** - 灵活的告警规则定义、版本管理、条件验证
- ✅ **回放控制** - 播放/暂停、速度调节(0.5x/1x/2x/4x)、逐帧前进/后退、时间跳转
- ✅ **人工确认** - 对活动告警进行人工确认，添加操作员备注
- ✅ **撤销确认** - 支持撤销误操作，保持操作可追溯
- ✅ **会话恢复** - 自动持久化到localStorage，重启后恢复完整状态
- ✅ **场景快照** - 手动命名保存多节点，含完整事件、游标、告警、确认、备注、规则；一键恢复、导出JSON、导入共享
- ✅ **回放导出** - 导出带处理状态的完整时间轴
- ✅ **乱序处理** - 智能处理重复事件、早到清除事件等异常场景
- ✅ **无依赖运行** - 纯前端应用，无需后端，不依赖地图或遥测平台

## 界面展示

- **当前游标** - 紫色高亮线显示当前回放位置，支持拖拽跳转
- **活动告警** - 按级别(严重/重要/一般)展示当前活动告警，显示持续时间
- **未解决输入** - 展示已到达但未处理的原始事件，标记异常状态
- **确认历史** - 按时间倒序展示所有确认/撤销操作，支持二次撤销
- **场景快照** - 手动命名保存多节点，含完整6类数据；支持冲突前确认、恢复后撤销、导出/导入、跨刷新保留

## 主流程使用说明

### 1. 导入样例事件

系统启动后会自动加载内置的乱序样例事件。也可以点击「加载样例」按钮重新加载。

如需导入自定义事件：
1. 点击「导入」按钮
2. 选择符合格式的JSON文件
3. 系统自动按时间戳排序并重置回放

### 2. 配置告警规则

1. 点击左侧「告警规则」标签页
2. 点击「新建」创建新规则
3. 填写规则名称、版本、匹配条件(JavaScript表达式)、告警级别
4. 点击「验证条件」测试规则语法
5. 点击「保存」完成规则创建

### 3. 启动回放

1. 点击绿色播放按钮开始回放
2. 使用速度按钮调节回放速度(0.5x/1x/2x/4x)
3. 点击暂停按钮暂停回放
4. 使用左右箭头按钮逐帧前进/后退
5. 拖拽时间轴游标或在跳转框输入时间直接跳转到指定位置

### 4. 人工确认告警

1. 在「活动告警」面板查看当前活动告警
2. 点击告警卡片展开详情
3. 输入确认备注(可选)
4. 点击「确认告警」完成确认
5. 确认后可点击「撤销确认」撤销操作

### 5. 导出回放结果

1. 点击「导出」按钮
2. 系统自动下载包含完整状态的JSON文件
3. 导出内容包括：所有事件、告警状态、确认历史、游标位置、规则版本

##### 导入恢复（同事收到 JSON 文件后）

**操作**：
1. 点击快照面板右上角的灰色 **「⬆ 导入」** 按钮
2. 系统弹出原生文件选择器，默认只显示 `.json` 文件
3. 选择刚才导出的 JSON 文件

**你会看到（成功）**：
- 右上角弹绿色 Toast：`✅ 已导入快照 "演练节点1"`
- 列表新增一条快照，Tab 徽章数字 +1
- 如果本地已经有同名快照，**不会冲突**，新快照名称会自动加上后缀：
  - 原名：`演练节点1`
  - 导入后名：`演练节点1 (导入于 2026/6/22 13:05:12)`

**你会看到（失败 - 6 种异常）**：

| 异常情况 | 红色 Toast 显示 | 对当前会话的影响 |
|---------|---------------|---------------|
| 文件内容不是合法 JSON | `导入失败：解析失败: <JSON.parse 错误详情>` | **零污染，完全不受影响** |
| schemaVersion 不是 1（比如 v99） | `导入失败：不兼容的快照版本：期望 v1，实际 v99` | **零污染** |
| 缺少 schemaVersion 字段 | `导入失败：不兼容的快照版本：期望 v1，实际 vundefined` | **零污染** |
| 缺少 snapshot 字段 | `导入失败：快照数据缺失或损坏` | **零污染** |
| 必填字段缺失（如没有 cursor） | `导入失败：快照字段缺失: cursor` | **零污染** |
| events 是空数组 | `导入失败：快照事件数据无效` | **零污染** |

**零污染的意思**：导入失败时，`snapshots` 列表、游标位置、活动告警、确认历史 — **所有和当前回放相关的状态一个字节都不会变**。失败只是右上角弹个红 Toast，其他什么都不动。

---

#### 6.7 第 7 步：刷新或重开浏览器后仍然保留

**真实存储行为（不是手动导出那种，是浏览器自动帮你存的）**：
- **存储位置**：浏览器 `localStorage`，键名：`replay:snapshots`
- **触发写入时机（全部都是自动的，不用用户点什么）**：
  - 每次 `saveSnapshot()` 成功（新建 or 覆盖）→ 立刻重写整个 `replay:snapshots`
  - 每次 `deleteSnapshot()` 成功 → 立刻重写整个 `replay:snapshots`
  - 每次 `importSnapshot()` 成功 → 立刻重写整个 `replay:snapshots`
- **触发读取时机**：
  - 页面启动 / 刷新时，`loadSession()` 会从 `replay:snapshots` 读回整个列表
- **容量限制**：受浏览器 localStorage 限制（通常 5MB，每个快照 ~20KB，足够存几百个）

**复现步骤**：
1. 新建 2~3 个快照，记录各自的名称、游标位置
2. 刷新页面（按 F5 或关闭标签重开）
3. 页面加载完成后，点击「场景快照」Tab

**你会看到**：
- Tab 右侧数字徽章显示正确的快照数量
- 点击 Tab，所有快照卡片完整列出，和刷新前一模一样
- 任意点一张展开，游标、告警数、确认数和刷新前**完全一致**（字节级相同）
- 点击「恢复」按钮，可正常恢复到保存时的状态

## 乱序场景复现说明

本工具内置了三个典型乱序场景，用于验证系统的容错处理能力。

### 场景1：重复事件编号

**复现步骤：**

1. 确保已加载样例事件（点击「加载样例」）
2. 观察时间轴，找到 `evt-002` 事件
3. 该事件在 **5秒** 和 **15秒** 各出现一次（相同eventId）
4. 点击播放或逐帧前进，观察处理结果：
   - 5秒时：evt-002正常处理，触发CPU使用率告警
   - 15秒时：evt-002被标记为 `[重复]` 灰色标签，不触发新告警
   - 「未解决输入」面板显示"事件ID重复，已忽略"

**预期结果：**
- 第一个事件正常处理
- 重复事件被标记为 `duplicate` 状态
- 不产生新告警
- 回放不中断，继续推进

---

### 场景2：清除事件早于告警事件

**复现步骤：**

1. 确保已加载样例事件
2. 观察时间轴，找到 `evt-006` 和 `evt-007` 事件
3. `evt-006`(clear类型) 在 **3秒** 到达
4. `evt-007`(alert类型) 在 **18秒** 到达
5. 两个事件共享相同的 `correlationId: corr-disk-01`
6. 逐帧前进观察处理过程：
   - 3秒时：evt-006到达，被标记为 `[待匹配]` 橙色标签，进入等待队列
   - 「未解决输入」面板显示"正在等待匹配的告警事件..."
   - 18秒时：evt-007到达，自动匹配到等待中的evt-006
   - 生成的告警直接标记为 `cleared` 状态，不会出现在活动告警中

**预期结果：**
- clear事件先进入等待队列，不丢失
- 对应alert到达后自动匹配清除
- 告警生成即被标记为已清除
- 「未解决输入」面板显示匹配成功的状态标记

---

### 场景3：撤销不存在的确认

**复现步骤：**

1. 回放至任意活动告警出现
2. 点击告警卡片展开，点击「确认告警」
3. 在「确认历史」面板看到确认记录
4. 打开浏览器开发者工具(F12)
5. 在Console中执行：
   ```javascript
   // 尝试撤销一个不存在的确认ID
   const store = window.__REPLAY_STORE__; // 或直接操作
   store.getState().undoConfirmation('non-existent-id');
   ```
6. 或者：先撤销一次确认，然后尝试再次撤销同一条记录

**预期结果：**
- 系统弹出红色错误提示：「无法撤销：确认记录不存在」或「无法撤销：该确认已被撤销」
- 当前回放状态不受影响
- 确认历史不产生新记录
- 告警状态保持不变

---

### 场景4：异常输入保护

**复现步骤：**

1. 点击「导入」按钮
2. 选择一个损坏的JSON文件（或格式错误的文件）
3. 观察系统反应

**预期结果：**
- 显示错误提示信息
- 当前回放会话不受影响
- 可继续正常操作

## 会话恢复验证

**复现步骤：**

1. 进行一段时间的回放，完成以下操作：
   - 播放到中间位置
   - 确认几个告警
   - 添加操作员备注

## 数据格式说明

### 事件格式

```json
{
  "eventId": "evt-001",
  "timestamp": 1719000000000,
  "type": "alert",
  "source": "cpu-monitor",
  "title": "CPU使用率过高",
  "payload": {
    "node": "node-01",
    "cpuUsage": 92,
    "threshold": 80
  },
  "correlationId": "corr-cpu-01"
}
```

字段说明：
- `eventId` - 事件唯一标识，用于去重
- `timestamp` - 事件时间戳(毫秒)
- `type` - 事件类型: `alert`/`clear`/`info`
- `source` - 事件来源
- `title` - 事件标题
- `payload` - 原始事件数据，用于规则匹配
- `correlationId` - 关联ID，用于匹配alert和clear事件

### 规则条件表达式

规则条件使用JavaScript表达式，可以访问以下变量：
- `event.eventId` - 事件ID
- `event.timestamp` - 时间戳
- `event.type` - 事件类型
- `event.source` - 事件来源
- `event.title` - 事件标题
- `event.payload.*` - 自定义载荷字段

示例：
```javascript
// 匹配CPU使用率超过阈值的告警
source === 'cpu-monitor' && type === 'alert' && payload.cpuUsage > payload.threshold

// 匹配特定来源的严重告警
source === 'database' && type === 'alert' && payload.errorCount > 100
```

## 存储键说明

所有数据存储在浏览器 localStorage 中：

| 键名 | 内容 | 说明 |
|------|------|------|
| `replay:session` | 会话状态 | 游标位置、确认历史、操作员备注 |
| `replay:events` | 事件数据 | 导入的完整事件时间轴 |
| `replay:rules` | 告警规则 | 告警规则配置 |
| `replay:snapshots` | 场景快照列表 | 所有手动保存的命名快照 |
| `replay:lastExport` | 最后导出内容 | 上次导出的JSON内容 |
